# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Repository Overview

**Zunftgewerk** — enterprise multi-tenant SaaS monorepo. pnpm workspaces + Turborepo orchestrating three frontend apps and a Spring Boot backend.

| Package | Filter name | Port | Purpose |
|---|---|---|---|
| `apps/landing` | `@zunftgewerk/landing` | 3000 | Marketing, pricing, auth, onboarding, authenticated dashboard |
| `apps/web` | `@zunftgewerk/web` | 3001 | Tenant admin, license management, operations cockpit |
| `apps/mobile` | `@zunftgewerk/mobile` | — | Expo offline-first field client |
| `services/api` | (Gradle) | 8080 / gRPC 9090 | Spring Boot backend (modular monolith) |
| `packages/proto` | — | — | Protobuf contracts (shared by API and mobile) |

Full architecture docs live in `docs/architecture/`.

## Commands

### Frontend

```bash
# Install all deps
pnpm install

# Dev servers
pnpm dev:landing          # or: pnpm --filter @zunftgewerk/landing dev
pnpm dev:web              # or: pnpm --filter @zunftgewerk/web dev
pnpm dev:mobile           # Expo — opens in browser/simulator

# Build
pnpm --filter @zunftgewerk/landing build
pnpm --filter @zunftgewerk/web build

# Type checking
pnpm --filter @zunftgewerk/landing typecheck
pnpm --filter @zunftgewerk/web typecheck

# Build all packages
pnpm build                # runs via Turborepo
```

### Backend (Spring Boot)

```bash
cd services/api
gradle bootRun            # starts on :8080, gRPC on :9090
gradle test               # JUnit tests (uses H2 in-memory)
gradle testClasses        # compile test classes only (needed for e2e seed script)
```

### Infrastructure

```bash
docker compose -f infra/docker-compose.yml up -d    # Postgres 16 + Redis 7 + Mailpit
docker compose -f infra/docker-compose.yml down
# Mailpit web UI: http://localhost:8025
```

### E2E Tests (Playwright, `apps/web`)

```bash
# One-time setup: copy and fill in credentials
cp apps/web/.env.e2e.example apps/web/.env.e2e

# Seed the test user into a running Postgres instance
./scripts/e2e-seed-web-user.sh   # requires Docker container 'zunftgewerk-postgres'

# Run tests (auto-starts web dev server)
pnpm --filter @zunftgewerk/web test:e2e
pnpm --filter @zunftgewerk/web test:e2e:headed
pnpm --filter @zunftgewerk/web test:e2e:debug

# Run a single spec file
pnpm --filter @zunftgewerk/web exec playwright test e2e/auth-passkey-mfa.spec.ts
```

`apps/web/.env.e2e` requires `E2E_ADMIN_EMAIL`, `E2E_ADMIN_PASSWORD`, `E2E_ADMIN_TOTP_SECRET`. The seed script hashes the password with Argon2 and encrypts the TOTP secret using the same key as the API (`MFA_ENCRYPTION_KEY`). `PASSKEY_ORIGIN` on the API must match the `E2E_BASE_URL` origin exactly for WebAuthn to verify.

## Architecture

### Frontend Apps

Both Next.js 16 apps use **Tailwind v4** (CSS-first config via `@theme` in `globals.css`, no `tailwind.config.js`) and **App Router**. The `@/*` alias maps to the app root (no `src/` prefix in either app). No `middleware.ts` in either app.

**`apps/landing`** is the public-facing site. It owns all auth flows (login, forgot/reset password, onboarding wizard) and an authenticated dashboard. Key conventions:

- **API client** (`lib/api-client.ts`): `fetchApi(path, init)` — thin wrapper around `fetch` with `credentials: "include"`. Base URL via `NEXT_PUBLIC_API_URL` (defaults to `http://localhost:8080`).
- **Session detection** (`lib/session.ts`): `getSession()` is server-side only — calls `GET /v1/onboarding/status` with forwarded cookies (`cache: "no-store"`). Returns `Session | null` (no-throw). Used in the `(authenticated)` layout to redirect to `/login` before render.
- **Radix UI** is imported from the **unified `radix-ui` v1.4.3 package**, not `@radix-ui/react-*`
- **Static content** lives in `content/`; hooks in `hooks/`; shared primitives in `components/ui/`
- **Custom CSS classes** (`.billing-editorial-main`, `.premium-panel`, `.enterprise-grid`, `.onboarding-step-connector`, `.animate-panel-enter`, `.premium-noise`) are defined in `globals.css` with custom `@keyframes` — not generated by Tailwind
- **Theme** uses oklch color space in `@theme` block
- **Fonts**: Chakra Petch (`--font-display`) + IBM Plex Sans (`--font-body`) via `next/font/google`
- **Onboarding wizard** (`components/onboarding/`): 6-step flow (plan → account → verify → signin → billing → complete). URL-resumable via query params (`?plan=CODE&step=STEP_ID&verified=1&reason=MSG`). State managed by `useOnboardingFlow` hook.
- **Address autocomplete**: Proxied through Next.js API routes (`/api/address/autocomplete`, `/api/address/reverse`) to Nominatim (OpenStreetMap). Debounced 300ms. Filtered to DE/AT/CH.
- **Login form**: Two-step flow — credentials then MFA (if required). On success: `router.push("/dashboard"); router.refresh()`.
- **Loading states**: `loading.tsx` files export Skeleton-based screens matching final layout structure.
- **Button variants** (CVA): `default`, `destructive`, `outline`, `ghost`, `link`, `gradient`. The `gradient` variant uses `bg-linear-to-r` (Tailwind v4 syntax). `LoadingButton` wraps Button with `pending` prop.
- **Error display**: `GENERIC_ERROR_MESSAGE` constant in German. Uses `.destructive` color tokens.

**`apps/web`** runs on port 3001 and hosts the tenant admin dashboard:

- **API client** (`lib/api.ts`): `apiRequest<T>({path, method, body, token})` with Bearer JWT auth. Base URL via `NEXT_PUBLIC_API_BASE_URL` (defaults to `http://localhost:8080`).
- **Auth client** (`lib/auth-client.ts`): Uses `credentials: "include"` for cookie-based session. Functions: `login()`, `beginPasskey()`, `verifyPasskey()`, `verifyMfa()`, `refreshSession()`.
- **SignIn page** (`(auth)/signin`): 3-stage state machine (`CREDENTIALS` → `MFA` → `DONE`). Access token stored in `localStorage.zg_access_token`.
- **WebAuthn** (`lib/webauthn.ts`): Base64URL encode/decode, `createAssertion()` for auth, `createRegistration()` for enrollment. Uses `navigator.credentials` API directly.
- **Offline store** (`lib/crypto/offline-store.ts`): IndexedDB (`zunftgewerk-offline`) with AES-256-GCM encryption. Ephemeral session-scoped key in memory.
- **E2E tests**: Playwright with CDP virtual authenticator for passkey testing. TOTP helper with boundary-safe generation (waits if within 3s of 30s window). Manual .env parsing (no dotenv package).
- **Theme** uses HSL color space (different from landing's oklch).

**`apps/mobile`** is Expo 55 / React Native 0.84.1:

- `@/*` alias maps to `src/`. Bundle IDs: `com.zunftgewerk.mobile` (iOS + Android).
- **Encrypted storage** (`src/storage/encryptedDb.ts`): SecureStore key alias `zunftgewerk_device_key_ref`, keychainService `zunftgewerk.keys`. SQLCipher interface defined, native binding pending.
- **Sync client** (`src/sync/syncClient.ts`): Push/pull with vector clocks via `SyncTransport` interface. `runSyncCycle()` orchestrates push-then-pull.
- Currently a minimal scaffold — HomeScreen with placeholder UI.

### Backend (Spring Boot)

`services/api` is a **modular monolith** under `com.zunftgewerk.api`. Java 21, Spring Boot 3.3.6, Gradle 9.3.1.

| Module | Responsibility |
|---|---|
| `identity` | Users, auth (JWT RS256, Argon2id, passkeys via Yubico, TOTP MFA), sessions |
| `tenant` | Tenants, memberships, roles, workspace management |
| `plan` | Plan catalog (Free/Starter/Professional), subscriptions |
| `billing` | Stripe integration, webhook processing with retry + dead-letter queue |
| `license` | Devices, seat licensing, entitlements |
| `sync` | Vector-clock aware client sync with conflict resolution |
| `audit` | Immutable append-only audit event store |
| `onboarding` | Multi-step onboarding progress (public endpoint) |

Shared infrastructure lives in `config/` (security, CORS, Redis, JPA) and `shared/` (tenant context holders, JWT service).

**gRPC contracts** are defined in `packages/proto/zunftgewerk/v1/` (auth, common, license, plan, sync, tenant). The API build pulls these via a `sourceSets` override in `build.gradle.kts` pointing to `../../packages/proto`.

### Implementation Conventions (Backend)

- **Constructor injection only** — no field injection anywhere
- **Java records** for result types: `LoginResult`, `RefreshResult`, `MfaEnrollmentResult`, etc.
- **`@Transactional`** on service methods that modify state
- **JpaRepository interfaces only** — no custom repository implementations
- **Entities are standalone** — no shared base entity class. Each has manual UUID `@Id`, `@Column` constraints, POJO getters/setters
- **HTTP responses**: `ResponseEntity.ok(Map.of(...))` for success, `ResponseEntity.badRequest().body(Map.of("error", message))` for errors
- **Naming**: `*Service`, `*Controller`, `*Entity`, `*Repository`, `*Result` (record), `*Config`, `*HttpRequest` (record)
- **Cookie name**: `zg_refresh_token` (set by `AuthCookieService`)
- **Cookie-based auth in controllers**: `peekUser(cookieHeader)` via `RefreshTokenService` extracts session without requiring authentication. Admin check queries `MembershipRepository` for `owner`/`admin` role.
- **Cookie extraction**: Manual string parsing of `Cookie` header (not Spring's `@CookieValue`)
- **Rate limiting**: Redis-backed, key pattern `zg:auth:ratelimit:{action}:{identifier}:{fingerprint}`. Fingerprint from `X-Forwarded-For` or `remoteAddr`. Gracefully disabled if Redis is unavailable.
- **Token generation**: 48 random bytes → Base64 URL-encoded (no padding) for refresh tokens
- **Password hashing**: Argon2id (time=3, memory=65536KB, parallelism=1)
- **Token hashing**: SHA-256 hex for all stored tokens (refresh, email verification, password reset)

### Auth Flow Summary

1. **Login** → `POST /v1/auth/login` → returns `AUTHENTICATED` (with tokens) or `MFA_REQUIRED` (with mfaToken)
2. **MFA** → `POST /v1/auth/mfa/verify` → verifies TOTP (±1 window) or backup code → returns full tokens
3. **Passkey** → `POST /v1/auth/passkey/begin` + `/verify` → WebAuthn challenge/response → same as login
4. **Refresh** → `POST /v1/auth/refresh` → rotates refresh token (same familyId), detects reuse → revokes entire family on reuse
5. **Session peek** → `GET /v1/onboarding/status` → reads cookie without auth, returns onboarding progress

MFA enforcement: admin/owner roles require MFA; member roles skip MFA.

### Data Model Invariants

- Every tenant-owned entity carries `tenant_id` with `ON DELETE CASCADE`
- Schema migrations are Flyway V1–V8 (`src/main/resources/db/migration/`)
- Client operations tracked for idempotency (sync module, `client_op_id`)
- Subscriptions: 1:1 with tenant (UNIQUE constraint on `tenant_id`)
- All timestamps use `TIMESTAMPTZ`
- UUIDs for all primary keys (not SERIAL)

### Feature Flags

Referenced in codebase but currently hardcoded in logic (no runtime toggle file). The flags mentioned: `stripeBilling`, `passkeyAuth`, `mfaEnforcementAdmin`, `authV2Enabled`, `syncVectorResolverEnabled`.

## Environment Variables

Copy `.env.example` at the repo root — it documents every variable. Key ones:

| Variable | Where | Notes |
|---|---|---|
| `NEXT_PUBLIC_API_URL` | `apps/landing` | API base URL (default: `http://localhost:8080`) |
| `NEXT_PUBLIC_API_BASE_URL` | `apps/web` | API base URL (default: `http://localhost:8080`) |
| `DATABASE_URL` / `_USERNAME` / `_PASSWORD` | API | PostgreSQL (default: `zunftgewerk`/`zunftgewerk`) |
| `JWT_PRIVATE_KEY_PEM` / `JWT_PUBLIC_KEY_PEM` | API | RS256 key pair (auto-generated if empty) |
| `STRIPE_SECRET_KEY` / `STRIPE_WEBHOOK_SECRET` | API | Stripe integration |
| `PASSKEY_RP_ID` / `PASSKEY_ORIGIN` | API | Must match web app origin exactly |
| `MFA_ENCRYPTION_KEY` | API + seed script | AES-128-GCM TOTP encryption |
| `AUTH_COOKIE_SAMESITE` | API | `Lax` (default), `None`, or `Strict` |
| `REDIS_HOST` / `REDIS_PORT` | API | Rate limiting + session cache |
| `OPENROUTESERVICE_API_KEY` | Landing | Geocoding (proxied via API routes) |
