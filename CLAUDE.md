This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Repository Overview

**Zunftgewerk** — enterprise multi-tenant SaaS monorepo. pnpm workspaces + Turborepo orchestrating three frontend apps and a Spring Boot backend.

| Package | Filter name | Port | Purpose |
|---|---|---|---|
| `apps/landing` | `@zunftgewerk/landing` | 3000 | Marketing, pricing, auth, onboarding |
| `apps/web` | `@zunftgewerk/web` | 3001 | Tenant dashboard + admin |
| `apps/mobile` | `@zunftgewerk/mobile` | — | Expo field client |
| `services/api` | (Gradle) | 8080 / gRPC 9090 | Spring Boot backend |
| `packages/proto` | — | — | Protobuf contracts (shared by API and mobile) |

## Commands

### Frontend

```bash
# Install all deps
pnpm install

# Dev servers
pnpm dev:landing          # or: pnpm --filter @zunftgewerk/landing dev
pnpm dev:web              # or: pnpm --filter @zunftgewerk/web dev
pnpm dev:mobile           # Expo — opens in browser/simulator

# Build
pnpm --filter @zunftgewerk/landing build
pnpm --filter @zunftgewerk/web build

# Type checking
pnpm --filter @zunftgewerk/landing typecheck
pnpm --filter @zunftgewerk/web typecheck

# Build all packages
pnpm build                # runs via Turborepo
```

### Backend (Spring Boot)

```bash
cd services/api
gradle bootRun            # starts on :8080, gRPC on :9090
gradle test               # JUnit tests (uses H2 in-memory)
gradle testClasses        # compile test classes only (needed for e2e seed script)
```

### Infrastructure

```bash
docker compose -f infra/docker-compose.yml up -d    # Postgres + Redis
docker compose -f infra/docker-compose.yml down
```

### E2E Tests (Playwright, `apps/web`)

```bash
# One-time setup: copy and fill in credentials
cp apps/web/.env.e2e.example apps/web/.env.e2e

# Seed the test user into a running Postgres instance
./scripts/e2e-seed-web-user.sh   # requires Docker container 'zunftgewerk-postgres'

# Run tests (auto-starts web dev server)
pnpm --filter @zunftgewerk/web test:e2e
pnpm --filter @zunftgewerk/web test:e2e:headed
pnpm --filter @zunftgewerk/web test:e2e:debug

# Run a single spec file
pnpm --filter @zunftgewerk/web exec playwright test e2e/auth-passkey-mfa.spec.ts
```

`apps/web/.env.e2e` requires `E2E_ADMIN_EMAIL`, `E2E_ADMIN_PASSWORD`, `E2E_ADMIN_TOTP_SECRET`. The seed script hashes the password with Argon2 and encrypts the TOTP secret using the same key as the API (`MFA_ENCRYPTION_KEY`). `PASSKEY_ORIGIN` on the API must match the `E2E_BASE_URL` origin exactly for WebAuthn to verify.

## Architecture

### Frontend Apps

Both Next.js apps use **Tailwind v4** (CSS-first config via `@theme` in `globals.css`, no `tailwind.config.js`) and **App Router**. The `@/*` alias maps to the app root (no `src/` prefix in either app).

**`apps/landing`** is the public-facing site. It owns all auth flows (login, forgot/reset password, onboarding wizard). Forms POST to the backend API via `lib/api-client.ts` (`NEXT_PUBLIC_API_URL`, defaults to `http://localhost:3001` — note: that's the web app port; override with the API URL in production). Key landing-specific conventions:
- Radix UI is imported from the **unified `radix-ui` v1.4.3 package**, not `@radix-ui/react-*`
- Static content lives in `content/`; hooks in `hooks/`; shared primitives in `components/ui/`
- CSS utility classes like `.billing-editorial-main`, `.premium-panel`, `.enterprise-grid`, `.onboarding-step-connector` are defined in `globals.css` — not generated by Tailwind

**`apps/web`** runs on port 3001 and hosts the tenant dashboard. It references the API via `NEXT_PUBLIC_API_BASE_URL`.

**`apps/mobile`** is Expo 55 / React Native 0.84.1. The `@/*` alias maps to `src/`. Uses `expo-secure-store` for credential storage.

### Backend (Spring Boot)

`services/api` is a **modular monolith** under `com.zunftgewerk.api`. Each business domain is a self-contained module:

| Module | Responsibility |
|---|---|
| `identity` | Users, auth (JWT RS256, Argon2, passkeys, TOTP MFA), sessions |
| `tenant` | Tenants, memberships, roles |
| `plan` | Plan definitions, feature/limit config |
| `billing` | Stripe integration, subscription lifecycle, webhook processing |
| `license` | License assignment and entitlements |
| `sync` | Vector-clock aware client sync with conflict metadata |
| `audit` | Immutable append-only audit event store |

Shared infrastructure lives in `config/` (security, CORS, Redis, JPA) and `shared/` (base entities, tenant context).

**gRPC contracts** are defined in `packages/proto/zunftgewerk/v1/` (auth, common, license, plan, sync, tenant). The API build pulls these via a `sourceSets` override in `build.gradle.kts`.

### Data model invariants
- Every tenant-owned entity carries `tenant_id`
- Schema migrations are Flyway (`src/main/resources/db/migration/`)
- Client operations are tracked for idempotency (sync module)

### Feature flags
Runtime toggles live in `config/feature-flags.json`. Key flags: `stripeBilling`, `passkeyAuth`, `mfaEnforcementAdmin`, `authV2Enabled`, `syncVectorResolverEnabled`.

## Environment Variables

Copy `.env.example` at the repo root — it documents every variable. Key ones:

| Variable | Where | Notes |
|---|---|---|
| `NEXT_PUBLIC_API_URL` | `apps/landing` | API base URL |
| `NEXT_PUBLIC_API_BASE_URL` | `apps/web` | API base URL |
| `DATABASE_URL` / `_USERNAME` / `_PASSWORD` | API | PostgreSQL |
| `JWT_PRIVATE_KEY_PEM` / `JWT_PUBLIC_KEY_PEM` | API | RS256 key pair |
| `STRIPE_SECRET_KEY` / `STRIPE_WEBHOOK_SECRET` | API | Stripe |
| `PASSKEY_RP_ID` / `PASSKEY_ORIGIN` | API | Must match web app origin |
| `MFA_ENCRYPTION_KEY` | API + seed script | AES-128-GCM TOTP encryption |
